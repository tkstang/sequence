  - Requirement: FR1

### Important

- **I1: Invite login-first path drops the invite** (`apps/web/src/app/login/page.tsx:34`)
  - Issue: The join page offers the required login-first path with `href="/login?next=/join/{code}"` (`apps/web/src/app/join/[code]/join-view.tsx:157`), but successful login always executes `router.replace('/dashboard')`. A registered visitor who chooses "log in" from an invite lands on the dashboard instead of returning to the invite and cannot complete the documented login-first join flow.
  - Fix: Read and sanitize a relative `next` search param in the login page, then redirect there after successful sign-in. Consider the same helper for signup if signup becomes an invite path. Add a test for `/login?next=/join/abc123`.
  - Requirement: FR3, FR1

- **I2: No-winner FFA concedes still show false losses in dashboard/history lists** (`packages/api/src/game/routes/my-games.ts:106`)
  - Issue: The carried `head-to-head.ts` fix correctly uses the `GameConceded` event so non-conceding teams in a no-winner FFA concede record neither win nor loss. However, p05's list payloads still expose only `won: g.winnerTeam !== null && g.winnerTeam === mine.team` in `game.myGames`, and `history.myGames` has the same binary calculation at `packages/api/src/history/routes/my-games.ts:107`. The dashboard/history UI then renders `L` for every `won: false` row, so non-conceding users see a false loss in recent results and completed games.
  - Fix: Return a tri-state result such as `result: 'win' | 'loss' | 'none'` or `won: boolean | null` from both list queries. For `winnerTeam === null && endReason === 'concede'`, join/read `GameConceded.payload.team` and mark only the conceding team as a loss; mark non-conceding teams as neutral/no-result. Update DashboardView/HistoryView rendering and add integration/component tests for a no-winner FFA non-conceder.
  - Requirement: FR11, FR14

### Medium

- **M1: Dashboard CTAs nest buttons inside links** (`apps/web/src/app/dashboard/dashboard-view.tsx:114`)
  - Issue: The Create and Pass-and-play CTAs render `<Link><Button /></Link>`, which becomes an anchor containing a button. That is invalid interactive nesting and undermines the p05 a11y claim for the dashboard's primary actions. It may still navigate in common browsers, but keyboard and assistive-tech behavior is not reliable.
  - Fix: Render the CTA as a styled `Link`, or extend the button primitive with an `asChild`/link variant so only one interactive element is emitted. Add a regression check that the CTAs are links without nested buttons.
