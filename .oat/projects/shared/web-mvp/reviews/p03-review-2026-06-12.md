### Important

- **I1: `SameSite=Lax` session cookies break cross-site prod auth** (`packages/api/src/user/auth.ts:53-59`)
  - Issue: `advanced.defaultCookieAttributes` sets `sameSite: 'lax'`. Verified live: `Set-Cookie: better-auth.session_token=…; HttpOnly; Secure; SameSite=Lax`. In production the web app (Vercel domain) and API (Railway domain) are different *sites*, and `Lax` cookies are not attached to cross-site `fetch`/XHR or WebSocket upgrade requests. Login will appear to succeed, but every credentialed tRPC call and the WS upgrade will arrive anonymous. Local dev masks this because `localhost:3000` → `localhost:3001` is same-site (port is ignored for SameSite).
  - Fix: drive cookie attributes by env — production: `sameSite: 'none'`, `secure: true` (consider `partitioned` for future-proofing), or serve web+API under one registrable domain (custom domain on both, `crossSubDomainCookies`). Apply the same attributes to the p04 guest cookie when it is issued. Add a p07 smoke assertion that a credentialed call succeeds cross-origin.
  - Requirement: FR1 (sessions validated on the API server, from the web client), FR6 (WS auth); design §Security Considerations, §CORS.
  - Timing: fix any time before p07-t02/t03; nothing in p03/p04 local testing is affected.

- **I2: Missing `trustProxy` → per-IP rate limiting is ineffective/DoS-prone in production** (`packages/api/src/server.ts:49-56`; `packages/api/src/shared/rate-limit-middleware.ts:30-35`)
  - Issue: Design §Threat Mitigation requires auth/preview/join "rate-limited per IP" (NFR5). `@fastify/rate-limit`'s default key is `request.ip`; the Fastify instance is built without `trustProxy`, so behind Railway's edge proxy `request.ip` is the proxy address — all users share one 20/min bucket on `/api/auth/*` (any single client can lock out login for everyone), and per-IP enforcement is meaningless. Separately, the tRPC limiter's `defaultKey` trusts the raw `x-forwarded-for` header, which is client-spoofable when the API is reached directly (limit evasion).
  - Fix: set `trustProxy: true` (env-gated for production) on the Fastify factory so `request.ip` resolves from `X-Forwarded-For` set by the trusted edge; thread `request.ip` into the tRPC `Context` and key the limiter on it instead of parsing `x-forwarded-for` manually.
  - Requirement: NFR5; design §Threat Mitigation.
  - Timing: code change is in p03 files; can land as a follow-up task before p04-t03 attaches the limiter to `preview`/`join`.

### Medium

- **M1: Malformed guest cookie → 500 instead of FORBIDDEN** (`packages/api/src/trpc.ts:108`)
  - Issue: `readCookie` calls `decodeURIComponent(...)` on the cookie value; a cookie like `sequence_guest=%` throws `URIError`, which escapes `resolveSeat` as `INTERNAL_SERVER_ERROR`. An anonymous caller can trigger 500s (and error-log noise) at the authorization boundary; correct semantics are "not a participant" → FORBIDDEN. Verified `decodeURIComponent('%')` throws.
  - Fix: wrap the decode in try/catch and return `null` on failure (or fall back to the raw value).

- **M2: `games.rematch_of` has no DB-level self-FK** (`packages/api/src/db/schema/games.ts:76`)
  - Issue: The recorded deviation (implementation.md) justifies plain-text `created_by`/`user_id` columns by Better Auth migration ordering — sound. But `rematch_of` references `games.id`, an api-owned table in the same migration, where that rationale does not apply; implementation.md explicitly states "the `game_id` FKs (within api-owned tables) ARE enforced." The p04-t12 expiry sweep deletes games, which can leave dangling `rematch_of` pointers.
  - Fix: additive migration adding `rematch_of REFERENCES games(id) ON DELETE SET NULL` (before/with p04-t12), or extend the Deviations row to cover `rematch_of` deliberately.

### Minor

- **m1: Auth bridge rejects non-JSON bodies with 415** (`packages/api/src/server.ts:87-106`)
  - Issue: Fastify only parses `application/json`/`text/plain` by default; a POST to `/api/auth/*` with `application/x-www-form-urlencoded` gets 415 before the handler runs. Email+password flows are JSON and GitHub/Google callbacks are GET, so nothing breaks today — but an OAuth provider using `form_post` (e.g. Apple) would.
  - Suggestion: note the constraint, or register a pass-through parser for form bodies on the auth route when social providers are configured.
- **m2: Duplicated auth bridge in harness** (`packages/api/src/test/harness.ts:89-108`)
  - Issue: The harness re-implements `toWebRequest` + response streaming inline. Two copies of subtle bridge logic can drift (e.g. only one gets a future cookie/header fix).
  - Suggestion: export `toWebRequest` (and a small `sendWebResponse`) from `server.ts` and reuse.
- **m3: Rate limiter never evicts stale buckets** (`packages/api/src/shared/rate-limit-middleware.ts:50-68`)
  - Issue: `buckets` retains one entry per distinct key forever; on a long-lived process this grows unbounded with distinct IPs.
  - Suggestion: delete the key when the filtered window is empty, or sweep periodically. Fine at MVP scale; cheap to fix now.
- **m4: Weak-secret floor** (`packages/api/src/env.ts:16`)
  - Issue: `BETTER_AUTH_SECRET: z.string().min(1)` accepts a 1-char HMAC/signing secret.
  - Suggestion: `min(32)` to fail fast on misconfiguration (it also signs guest tokens).
- **m5: Local-game seat fallback is insertion order** (`packages/api/src/trpc.ts:175`)
  - Issue: Doc comment promises "the lowest seat" when `currentSeat` is unset, but `seats[0]` from an unordered `SELECT` is arbitrary row order.
  - Suggestion: add `orderBy(gamePlayers.seat)` or fix the comment. Only reachable for a pre-start local game (local games skip the lobby, so today it is near-unreachable).

### Surviving / new findings

- **I3 (Important, surviving residual of I2): `ctx.ip` is `'unknown'` on the WebSocket path — WS callers share one rate-limit bucket** (`packages/api/src/trpc.ts:88`; `packages/api/src/server.ts:131-135`)
  - Issue: For HTTP, `createContext` receives the FastifyRequest and `req.ip` honors `trustProxy` — correct. But for WS, the fastify adapter calls `onConnection(socket, req.raw)` (verified in `@trpc/server@11.17.0/dist/adapters/fastify/index.mjs`), so `createContext` receives the bare `IncomingMessage`, which has no `.ip` → the `req.ip ?? 'unknown'` fallback yields `'unknown'` for every WS connection. tRPC over WS serves queries/mutations too, so once p04-t03 attaches the limiter to `preview`/`join`, all anonymous WS callers share a single `ip:unknown` bucket — one client can lock out guest preview/join for everyone on the WS path (the original I2 lockout, resurfaced for WS), and per-IP isolation is void there.
  - Fix: stamp the Fastify-resolved IP onto the raw request before the WS handler runs — e.g. `app.addHook('onRequest', async (req) => { (req.raw as IncomingMessage & { clientIp?: string }).clientIp = req.ip; })` — and read `req.ip ?? req.clientIp ?? 'unknown'` in the context factory (the WS `req` *is* the stamped `req.raw`). Alternatively fall back to `req.socket?.remoteAddress` (loses trusted-XFF resolution behind the proxy but removes the shared bucket). Add a WS-path assertion.
  - Timing: before p04-t03 attaches the limiter (same window the original I2 specified). No p03 exit criterion is affected — nothing consumes `ctx.ip` in p03.

- **M3 (Medium, new): boolean `trustProxy: true` keys on the leftmost XFF entry — client-spoofable if Railway's edge appends rather than replaces** (`packages/api/src/server.ts:58,67`; `packages/api/src/env.ts:30-38`)
  - Issue: Verified empirically: with `trustProxy: true` and `X-Forwarded-For: 6.6.6.6, 7.7.7.7`, `request.ip` is `6.6.6.6` (leftmost = client-suppliable); with `trustProxy: 1` it is `7.7.7.7` (the hop the trusted edge appended). If Railway's edge *appends* to a client-supplied XFF instead of overwriting it, a prod attacker rotates fake leftmost entries to evade per-IP limits and frame arbitrary IPs. The env schema only admits `true/false/1/0`-as-boolean, so the operator cannot express the robust hop-count form. (Dev/test default-off is correct: socket address, unspoofable.)
  - Fix: prefer hop-count semantics (`trustProxy: 1` numeric) for the proxied prod deploy, or keep boolean but add a p07 smoke assertion that a forged client XFF does not alter the limiter key on Railway. Extending `TRUST_PROXY` to accept a hop count is a few lines in `env.ts`.
  - Timing: prod-only; bundle with the p07 deploy checks (same bucket as I1).
