## Summary

The engine is well-structured, pure, and broadly rule-faithful: corner-as-wild, jack semantics, dead-card resurrection, natural-over-jack consumption, win thresholds, turn loop, and the lifecycle state machine all check out against `rules-and-flows.md`, with a genuinely strong 112-test suite and seeded full-game simulations. However, the implementer's flagged advisory area (multi-sequence corner cases) hides real defects: `resolveSequenceChoice` does not enforce the one-reused-space rule (an illegal second sequence — and an illegal **win** — is grantable, verified by repro), and a >5 run in one direction silently discards a completed exactly-5 sequence in a crossing direction. `createGame` also rejects 6-player 2-team (3v3) games that p04-t05 explicitly requires, and ignores the seat/team mapping in its `PlayerSeed` inputs. Verdict: **fail** — fix before merge into the p03 integration line.

### Critical

- **C1: `resolveSequenceChoice` accepts a choice window that reuses two or more locked cells** (`packages/game-logic/src/apply-move.ts:173-193`)
  - Issue: `isValidChoice` checks only "exactly 5, contiguous in the eligible run, includes the placed cell." It never applies the reuse rule that `detectSequences` enforces on the exactly-5 path (`lockedCount <= 1`, `sequence-detection.ts:105`). A run extended through an existing own locked sequence (e.g. locked seq at row 4 cols 0–4, new chips at cols 5–6, place at col 7 → 8-run) lets the placer choose cols 3–7 — **two** cells reused from the first sequence — and the engine locks it as a fresh sequence and, in a 2-team game, declares `GameWon`. Verified by repro: the illegal window is accepted and `winner=1` is set. This violates FR5's "one reusable space between own sequences" (P0) with a game-deciding, user-triggerable outcome (any client can submit an arbitrary cell set to the p04 `chooseSequenceCells` route).
  - Fix: in `isValidChoice` (or in `resolveSequenceChoice` before locking), count cells in the chosen window already carrying `lockedBy` for sequences owned by `pending.team` and reject with `invalid-sequence-choice` when the count exceeds 1. Add a regression test mirroring the repro above.
  - Requirement: FR5 (sequence reuse rule), NFR1 (server-authoritative integrity).

### Important

- **I1: `detectSequences` returns only `choiceRequired` and discards completed exactly-5 sequences in other directions** (`packages/game-logic/src/sequence-detection.ts:90-131`)
  - Issue: one placement can complete an exactly-5 sequence vertically while creating a 6-run horizontally. The loop collects the 5-run into `autoLock`, then the `longest` >5 run short-circuits to `choiceRequired`, and `applyMove` (`apply-move.ts:304-318`) never locks the auto-lock candidate — the player permanently loses an earned sequence, and a legitimate double-sequence instant win (FR5, P0) becomes a single sequence. Verified by repro: after resolving the 6-run choice, `sequences.length === 1`, no winner. Same structural hole: two directions both >5 keeps only the longest; the other long run's choice is lost. `DetectionResult` cannot even represent "auto-lock + choice" simultaneously.
  - Fix: extend `DetectionResult` (and `PendingChoice`) so auto-lock sequences and a pending choice can coexist — e.g. `{ kind: 'choiceRequired', autoLock: Position[][], runLength, cells }`; in `applyPlace`, lock the auto-lock sequences immediately (win check included), then freeze on the remaining choice. Cover: 5-run crossing a 6-run; two crossing 6-runs.
  - Requirement: FR5 (double-sequence instant win; sequence detection), implementer-flagged advisory area.

- **I2: 6-player 2-team (3v3) games are rejected** (`packages/game-logic/src/create-game.ts:29-32`)
  - Issue: `teamCountFor` hardcodes `6 → 3 teams`. Official rules and plan p04-t05 ("validates team evenness — 6p: 3v3 or 2x3 per settings") both require 6 players as **either** 2 teams of 3 or 3 teams of 2. Verified by repro: seeds `[1,2,1,2,1,2]` (3v3) throw `uneven teams: expected 3 equal teams`. Win threshold would also differ (2 teams → 2 sequences), so this isn't recoverable host-side.
  - Fix: derive team count from the distinct teams in `players` (validate it's 2 or 3 and legal for the player count: 2p→2, 3p→3, 4p→2, 6p→2|3) instead of hardcoding per player count. `checkWin` already derives from `state.teams`, so it follows automatically.
  - Requirement: FR4 (team formation), plan p04-t05.

- **I3: `createGame` ignores the `PlayerSeed` seat/team mapping** (`packages/game-logic/src/create-game.ts:38-67`)
  - Issue: `alternatingOrder` only counts team sizes, then emits the positional pattern `[1,2,1,2,…]`. Verified by repro: seeds `[{seat:0, team:2},{seat:1, team:1}]` produce `teams: [1,2]` — seat 0 silently moved to team 1; seeds with teams `{1,3}` are accepted and silently remapped to `{1,2}`. When the p04 lobby passes real team selections, players get wrong teams/colors with no error. `p.seat` is likewise never read (array order is what matters), inviting drift between lobby seating and engine seating.
  - Fix: honor the seeds — validate that team ids are contiguous from 1 and that the seat-ordered team list alternates (or sort the seeds into alternating order and return the seat assignment explicitly); throw on any mismatch rather than silently rewriting.
  - Requirement: FR4 (self-sorted teams, derived turn order), plan p02-t03.

- **I4: No actor-seat seam — `not-your-turn` is unreachable and turn ownership is unenforceable in the engine** (`packages/game-logic/src/types.ts:156-169`, `apply-move.ts:63`)
  - Issue: `Move` carries no seat and `applyMove` assumes the mover is `state.currentSeat`, so the engine can never produce `not-your-turn` (the p02-t08 test at `apply-move.test.ts:61-74` concedes this by asserting `card-not-in-hand` instead). Plan p02-t08 RED requires the reducer to reject out-of-turn with `not-your-turn`, and p04-t07 expects "out-of-turn FORBIDDEN-by-engine (`not-your-turn` violation)" — impossible against this surface. Similarly `resolveSequenceChoice` takes no actor (the "only placer may resolve" rule of p04-t08 must be host-implemented), and `turnInDeadCard` accepts any seat at any time. Note: design.md §Component Design's `applyMove(state, move)` itself omits the seam, so this is part defect, part artifact drift — but the plan's RED specs and NFR1's "engine rejects illegal moves regardless of client behavior" point at the engine owning it.
  - Fix (preferred): add `seat: Seat` to `Move` (and an actor param to `resolveSequenceChoice`/`turnInDeadCard`); validate against `state.currentSeat`/`pendingChoice.seat` and return `not-your-turn`. Alternative: keep the surface, then update plan p04-t07/p04-t08 wording and design.md to delegate turn-ownership checks to the host explicitly — but the engine-owned check is cheap and serves the future offline host.
  - Requirement: NFR1, plan p02-t08/p04-t07/p04-t08; relates to the recorded `rng?` deviation (same interface block needs a design.md alignment note either way).

### Medium

- **M1: `turnInDeadCard` skips game/turn validation and allows unlimited turn-ins per turn** (`packages/game-logic/src/apply-move.ts:199-231`)
  - Issue: no `status === 'active'` check, no `pendingChoice` check, no current-seat check (any seat can swap mid-opponent-turn), and nothing limits turn-ins to one per turn — official rules and the default-mode analogue (`autoSwapDeadCard` swaps "at most one per turn") both cap at one. A hard-mode player with two dead cards can swap both (or swap while it isn't their turn) and the engine obliges.
  - Fix: add the three guards mirroring `applyMove`'s preamble plus `seat === state.currentSeat`; track a per-turn turned-in flag (cleared in `advanceTurn`) or document/enforce the one-per-turn cap host-side with a plan note.

- **M2: `validPlacements` defaults `team` to 1** (`packages/game-logic/src/display-helpers.ts:25-29`)
  - Issue: one-eyed jack targets are team-dependent. A team-2 caller that omits `team` gets team 1's targets — i.e. its **own** chips listed as removable — and every suggested removal would then be rejected by the reducer (`own-chip`), breaking the "agrees with applyMove" property the same file's tests assert. The design signature (`validPlacements(hand, board)`) predates the team param; a silent default is the worst of both.
  - Fix: make `team` required (callers in p06 always know their team). If a default must stay, omit one-eyed jacks from the result when `team` is not provided.

### Minor

- **m1: Unused/unreachable `RuleViolation` codes** (`packages/game-logic/src/types.ts:182-192`)
  - Issue: `not-your-turn` (see I4), `freed-cell-same-turn` (structurally enforced — removal ends the turn), `no-chip-to-remove`, and `not-a-wild-card` are never produced by any code path. Dead union members suggest enforced rules that aren't.
  - Suggestion: produce them where applicable (I4 covers `not-your-turn`) or remove/comment the structurally-enforced ones (`freed-cell-same-turn` deserves a comment pointing at the removal-ends-turn invariant).

- **m2: `DeadCardSwapped` for the incoming player is emitted before `TurnAdvanced`** (`packages/game-logic/src/apply-move.ts:392-412`)
  - Issue: the incoming player's auto-swap (start-of-their-turn semantics per rules-and-flows) lands in the event stream before the `TurnAdvanced` that announces their turn. Event-stream consumers reconstructing per-turn state see a hand mutation attributed to a turn that hasn't started.
  - Suggestion: push `TurnAdvanced` before running the auto-swap.

- **m3: `autoSwapDeadCard` silently discards the dead card when no replacement is drawable** (`packages/game-logic/src/dead-cards.ts:77-83`)
  - Issue: when deck and played pile are both empty (near-unreachable in practice), the hand shrinks with no event emitted — the event stream can no longer reconstruct hand size. Same pattern in `turnInDeadCard` (`apply-move.ts:219-221`).
  - Suggestion: either leave the dead card in hand when nothing can be drawn, or emit an event for the discard.
