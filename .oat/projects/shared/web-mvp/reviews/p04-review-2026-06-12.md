## Findings

### Critical

- **`version` is not obtainable by any client read path; `makeMove` is uncallable over pure tRPC** (`packages/api/src/shared/realtime/redaction.ts:64`, `packages/api/src/game/routes/on-game-event.ts:118`)
  - Issue: `makeMove` / `chooseSequenceCells` / `turnInDeadCard` all require `version: z.number().int().nonnegative()` (e.g. `make-move.ts:61`). The only place a client ever receives a current `version` is the **return value of its own prior successful mutation** (`move-engine.ts:183`). No query exposes it: `preview` (`preview.ts:68`), `myGames` (`my-games.ts:48`), and — critically — the subscription **snapshot** (`GameSnapshot`, `redaction.ts:64`; built in `on-game-event.ts:118`) all omit `version`, and broadcast events carry only `{seq,type,payload}` (`move-engine.ts:166`). `seq` (per-event) is not `version` (per-reduction), so it can't be derived. Result: a freshly-subscribed client, the next player to move, and the first mover after `start` (which returns only `{status,currentSeat}`, `start-game.ts:100`) have no way to learn the `version` needed to act. The CONFLICT path returns no current version either (error formatter surfaces only `ruleViolation`, `trpc.ts:160`). The phase passes its suite only because `full-game.e2e.test.ts:159` and the route tests read `games.version` straight from the DB (`h.db.select({version})`), and the Bruno `game` collection contains **no move request** at all (`bruno/game/` is signup/create/preview/set-team/my-games/history) — so the "fully playable over tRPC/Bruno with no UI" exit criterion is not actually exercised through the client-facing contract.
  - Fix: add `version` to `GameSnapshot` (populate from the `games` row in `loadGameState`/the subscription, alongside `currentSeat`) so the recovery snapshot is sufficient to act; and include the post-commit `version` in the broadcast envelope (or a dedicated event field) so non-actor clients track it as the turn advances. Optionally add the current `version` to the CONFLICT error `data` for a cheap resync. Add a Bruno `makeMove` request and an e2e assertion that drives a move using only client-reachable data (no direct DB read) to lock the contract.
  - Requirement: FR6 (real-time playability over the API), NFR1-adjacent recovery contract (design §Error Handling "full redacted snapshot … then the live stream").

### Important

- **Lifecycle transitions bypass the optimistic-concurrency version guard; a concurrent move can revert a concede/save/freeze** (`packages/api/src/game/routes/concede.ts:48`, `packages/api/src/game/routes/save-and-exit.ts:51`, `packages/api/src/game/presence.ts:97`, `packages/api/src/game/presence.ts:137`)
  - Issue: the move engine is the only writer that bumps + guards `games.version` (`state-mapping.ts:214`). `concede`, `saveAndExit`, and presence freeze/resume all `UPDATE games SET status=…` with **no version predicate and no version bump**. Under Postgres READ COMMITTED a real interleave loses the lifecycle write: T1 `makeMove` reads `version=5` and reduces; T2 `concede` commits `status='finished'` (version still 5); T1's persist `… WHERE version=5` still matches (concede never bumped it) and clobbers the row back to `status='active', version=6`. The concede is silently lost. `concede` is callable by **any** participant at any time (not just the current seat, `concede.ts:20`), so the current mover racing another player's concede/save is reachable in normal play, not a contrived case. The same shape lets a move revert a presence-driven freeze (`presence.ts:94`), leaving a game `active` with a departed player. No test covers a concede/save/freeze-vs-move race (only `make-move.test.ts:183` double-submit).
  - Fix: route these transitions through the same version protocol — read the current `version`, write with `… WHERE version = prev` and bump to `prev+1` (reuse `persistGameState`'s predicate pattern), mapping a zero-row update to CONFLICT — or take a `SELECT … FOR UPDATE` row lock at the top of each lifecycle transaction so it serializes against the move engine. Add a race test (concede vs concurrent move: exactly one outcome wins).

### Minor

- **3-team FFA concede records a loss against the non-conceding teams** (`packages/api/src/history/routes/my-record.ts:42`)
  - Issue: a 3-team concede sets `winnerTeam = null` (`concede.ts:46`); `myRecord` then counts `winnerTeam === null` as a loss for **every** participant, including the two teams that did not concede. The comment ("including no-winner FFA concedes") flags this as deliberate, but recording a loss for players who didn't concede is debatable and unspecified in FR11/FR14.
  - Suggestion: decide the intended FFA-concede semantics and encode it (e.g. only the conceding team's seats take the loss; others are excluded or recorded as a non-result). If the current behavior is intended, note it in the design history section so it isn't read as a bug later.

- **`history.myGames` cursor can skip rows on `finished_at` ties** (`packages/api/src/history/routes/my-games.ts:42`)
  - Issue: pagination orders by `desc(finishedAt)` and advances with `lt(finishedAt, cursor)`. Two games finishing in the same millisecond straddling a page boundary can be skipped. Friend-group scale makes this unlikely, but it is a latent correctness edge.
  - Suggestion: use a composite cursor (`finished_at`, `id`) for a total order, or accept and document the tie limitation.

- **Subscription snapshot omits `pendingChoice.placed` / `additionalRuns`** (`packages/api/src/shared/realtime/redaction.ts:117`)
  - Issue: a client recovering via snapshot mid-choice gets `{seat, cells}` but not the just-placed cell or queued runs. The server re-validates the window on `chooseSequenceCells` so this is not a correctness hole, but the placer's UI can't fully reconstruct the choice from a cold snapshot.
  - Suggestion: include `placed` (and, if useful to the UI, `additionalRuns`) in the snapshot's `pendingChoice` for parity with the live `PendingChoice` event.

- **Lingering lint warnings in scope** (`packages/api/src/game/routes/make-move.test.ts:86`, `packages/api/src/test/harness.ts:217`)
  - Issue: `oxlint` emits `consistent-function-scoping` warnings (non-blocking; the gate passes). Cosmetic only.
  - Suggestion: hoist the flagged helpers or leave as-is; no action required for merge.
