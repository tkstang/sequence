## Structural Conventions

- One file per game action in `src/game/routes/`, each exporting `<action>Route` built from `authedProcedure` / `gamePlayerProcedure` / `publicProcedure`.
- Inputs validated with zod at the procedure boundary; multi-write paths use `ctx.db.transaction`.
- Relative imports carry explicit `.ts` extensions; type-only imports use `import type`.
- Drizzle schema lives in `src/db/schema/`; SQL migrations are generated into `drizzle/` and committed (never hand-edited; `drizzle/` is lint/format-ignored).

## Behavioral Conventions

- **Redaction (NFR1):** never serialize another seat's hand or the deck to a non-owning recipient; use `redactEvent`/`buildSnapshot`. Local games are the sole exception (one connection owns all seats).
- **Version guard:** clients submit an expected game `version`; lifecycle mutations reject stale versions (optimistic concurrency).
- **Rate limiting:** key on `ctx.ip` only; never trust a hand-parsed `x-forwarded-for`. Production keeps `TRUST_PROXY=false`.
- **Realtime/timers/presence/limiter buckets are process-local** in the MVP ÔÇö do not assume horizontal scaling.
- Surface game-logic rule violations through the tRPC error formatter (`error.data.ruleViolation`), not string matching.

## Counter-Examples

- Adding a route that serializes full game state (other hands/deck) to a subscriber Ôćĺ privacy/NFR1 violation.
- A limiter keyed on a parsed `x-forwarded-for` header Ôćĺ forgeable when `TRUST_PROXY` is off.
- Hand-editing a file under `drizzle/` instead of running `drizzle-kit generate` Ôćĺ schema drift, failed predeploy migrate.
- A mutation that ignores the client `version` Ôćĺ lost-update/stale-write bug.
